  ### ✅ Python Function to Merge Overlapping Intervals:
  
  ```python
  def merge_intervals(intervals):
      # Sort the intervals based on the start time
      if not intervals:
          return []
  
      intervals.sort(key=lambda x: x[0])
      merged = [intervals[0]]
      
      for current in intervals[1:]:
          last = merged[-1]
          if current[0] <= last[1]:
              # Overlapping, merge them
              merged[-1] = [last[0], max(last[1], current[1])]
          else:
              # No overlap, add the current interval
              merged.append(current)
      
      return merged
  ```


  Implementing a **thread-safe singleton pattern** in C++ involves ensuring that only one instance of a class is created and that the singleton is safely accessible from multiple threads. The key challenge is to prevent **race conditions** during the instance creation and to ensure **memory visibility** between threads.
  
  ---
  
  ## ✅ Thread-Safe Singleton Pattern in C++
  
  Here is a standard and widely accepted implementation using **C++11 or later** (which provides `std::call_once` and `std::mutex`):
  
  ```cpp
  #include <mutex>
  #include <iostream>
  
  class Singleton {
  public:
      static Singleton& getInstance() {
          std::call_once(initOnce, &Singleton::initInstance);
          return *instance;
      }
  
      // Prevent copy and assignment
      Singleton(const Singleton&) = delete;
      Singleton& operator=(const Singleton&) = delete;
  
  private:
      Singleton() {
          std::cout << "Singleton created\n";
      }
  
      ~Singleton() {
          std::cout << "Singleton destroyed\n";
      }
  
      static void initInstance() {
          instance = new Singleton();
      }
  
      static std::once_flag initOnce;
      static Singleton* instance;
  };
  
  // Initialize static members
  std::once_flag Singleton::initOnce;
  Singleton* Singleton::instance = nullptr;
  ```
  
  ---
  
  ## 🔐 Thread Safety and Memory Barriers
  
  This implementation is **thread-safe** because:
  
  1. **`std::call_once`** ensures that the initialization code is executed only once, even in a multithreaded environment.
  2. **`std::once_flag`** is a synchronization object that ensures the initialization function is called exactly once.
  3. The `initInstance()` function is called only once, and the `instance` pointer is initialized only once.
  4. **Memory barriers** are implicitly enforced by `std::call_once`, ensuring that the memory writes (like `instance = new Singleton()`) are **visible to all threads** after the initialization is complete.
  
  ---
  
  ## 🧠 Memory Barrier Explanation
  
  In C++, a **memory barrier** (or **memory fence**) ensures that memory operations (reads and writes) are properly ordered across threads. This is crucial in concurrent programming to prevent **data races** and **undefined behavior**.
  
  In the context of the singleton pattern:
  
  - When `instance = new Singleton()` is executed, it is a **write** to a global pointer.
  - Without proper memory barriers, other threads may **see an uninitialized value** (e.g., a dangling pointer) before the constructor has completed.
  - `std::call_once` ensures that the initialization is done **atomically** and **sequentially consistent**, which **implicitly provides memory barriers** to ensure visibility of memory writes.
  
  ---
  
  ## 🧪 Example Usage
  
  ```cpp
  int main() {
      Singleton& s1 = Singleton::getInstance();
      Singleton& s2 = Singleton::getInstance();
  
      // s1 and s2 refer to the same instance
      std::cout << "Same object? " << (&s1 == &s2) << std::endl;
  
      return 0;
  }
  ```
  
  ---
  
  ## ✅ Summary
  
  | Feature | Description |
  |--------|-------------|
  | **Thread Safety** | Achieved using `std::call_once` and `std::once_flag` |
  | **Memory Barrier** | Provided implicitly by `std::call_once`, ensuring proper memory visibility |
  | **Lazy Initialization** | Instance is created only when first accessed |
  | **Singleton Pattern** | Only one instance is ever created |
  | **No Copy or Assignment** | Prevents multiple instances through deleted copy constructor and assignment operator |

  ## ✅ Alternative: Double-Check Locking (Not Recommended)
  
  While **double-check locking** is a classic approach, it **requires careful memory barrier usage** and is **error-prone**. The `std::call_once` approach is **simpler, safer, and more idiomatic** in C++11 and later.

  ```cpp
      }

      Singleton& operator=(const Singleton&) = delete;

  };

  ```
  
  ---
  
  ## ✅ Explanation
  
  ### 🔐 Thread Safety
  
  This implementation is **thread-safe** because:
  
  - A `std::mutex` is used to **synchronize access** to the `GetInstance()` method.
  - A `std::lock_guard` is used to **lock the mutex** when entering the function.
  - The `static` local variable `instance` inside `GetInstance()` is **initialized only once** due to the **static initialization** in C++11 and later.
  
  This is known as the **"Double-Check Locking"** pattern (though in this case, we're not doing double-checking explicitly, but the `std::lock_guard` ensures that the lock is held for the entire duration of the function call, making it thread-safe).
  
  ---
  
  ## 🧠 Memory Barrier Usage
  
  In C++, **memory barriers** (also known as **memory fences**) are used to **ensure that memory operations are ordered** correctly across threads. These are important in **concurrent programming** to prevent **data races** and **incorrect memory visibility**.
  
  ### 🔧 Memory Barriers in the Singleton Pattern
  
  In the above implementation, the compiler and CPU may perform **reordering** of memory operations for optimization. For example:
  
  - The compiler might **reorder** the construction of the singleton instance and the initialization of the `mutex_` (which is static).
  - The CPU might **reorder** memory accesses for performance.
  
  To prevent such reordering and ensure that the singleton is fully constructed **before** it is visible to other threads, we need to use **memory barriers**.

  ## ✅ Thread-Safe Singleton with Explicit Memory Barriers

  
  In modern C++ (C++11 and later), the **static local variable initialization** is **thread-safe** and **guaranteed to be properly ordered** by the C++ standard, which includes **memory barriers** implicitly. This means that the above implementation is **already thread-safe** without needing to manually insert memory barriers.
  
  ---
  
  ## ✅ When to Use Explicit Memory Barriers
  
  In **low-level** or **custom implementations**, you might need to insert **memory barriers** explicitly using:
  
  - `std::atomic_thread_fence(std::memory_order::seq_cst)`
  - `std::atomic_thread_fence(std::memory_order::acq_rel)`
  - `std::atomic_thread_fence(std::memory_order::release)`
  - `std::atomic_thread_fence(std::memory_order::acquire)`
  
  For example, in a **double-checked locking** implementation:
  
  ```cpp
  Singleton* Singleton::GetInstance() {
      if (!instance_) {
          std::lock_guard<std::mutex> lock(mutex_);
          if (!instance_) {
              instance_ = new Singleton();
              std::atomic_thread_fence(std::memory_order::release);
          }
      }
      std::atomic_thread_fence(std::memory_order::acquire);
      return instance_;
  }
  ```
  
  Here, `std::memory_order::release` ensures that the memory writes to `instance_` are visible to other threads, and `std::memory_order::acquire` ensures that the memory reads are ordered correctly.

  In Unix/Linux socket programming, **I/O multiplexing** is used to monitor multiple file descriptors (e.g., sockets) for I/O events (like read, write, or exception). Three common mechanisms for this are:
  
  - **`select()`**
  - **`poll()`**
  - **`epoll()`** (Linux-specific)
  - **`kqueue()`** (macOS and BSD-based systems)
  
  Let's compare **`epoll`**, **`kqueue`**, and **`select`**, focusing on their differences in functionality, performance, scalability, and usage.
  
  ---
  
  ## 🔹 1. `select()`
  
  ### Overview:
  - A system call that allows monitoring multiple file descriptors for I/O readiness.
  - Has been around since the early days of Unix.
  
  ### Key Features:
  - Works on all Unix-like systems (Linux, macOS, BSD, etc.).
  - Uses a **bitmask** to track the file descriptors.
  - Has a **limit** on the number of file descriptors it can monitor (usually 1024 or 2048, depending on the system).
  
  ### Syntax:
  ```c
  int select(int nfds, fd_set *readfds, fd_set *writefds, fd_set *exceptfds, struct timeval *timeout);
  ```
  
  ### Limitations:
  - **Slow performance** for large numbers of file descriptors due to **copying the entire set** of file descriptors in and out of kernel space.
  - **Inefficient** for monitoring many sockets.
  - **No persistent monitoring** — you must call `select()` every time you want to check for events.
  
  ---
  
  ## 🔹 2. `epoll` (Linux-specific)
  
  ### Overview:
  - Introduced in **Linux 2.6** as a **more efficient** I/O multiplexing mechanism.
  - Uses **file descriptors** to monitor other file descriptors (event-based).
  
  ### Key Features:
  - **Scalable** — can handle thousands of file descriptors efficiently.
  - Uses **edge-triggered** or **level-triggered** modes.
  - Uses an **event loop** to monitor file descriptors.
  - **No copying** of the file descriptor set — only the events are passed.
  
  ### Syntax (simplified):
  ```c
  int epoll_create(int size);
  int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
  int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);
  ```
  
  ### Advantages:
  - **High performance** for large numbers of file descriptors.
  - **No per-call copy** of file descriptors — only event data is copied.
  - **Persistent monitoring** — you can add/remove file descriptors dynamically.
  - **Supports edge-triggered (ET) and level-triggered (LT)** modes.
  
  ### Use Case:
  - Ideal for **high-performance servers** (e.g., web servers, chat servers, etc.)
  - Used in **Nginx**, **Redis**, and many other high-throughput applications.
  
  ---
  
  ## 🔹 3. `kqueue` (macOS, BSD, etc.)
  
  ### Overview:
  - Introduced in **BSD systems** (macOS, FreeBSD, NetBSD, OpenBSD).
  - Similar to `epoll` in functionality but with a different API.
  
  ### Key Features:
  - **Kernel-based** event notification system.
  - Uses **kqueue** as a file descriptor that you can monitor.

  - Used in **macOS**, **FreeBSD**, and other BSD-based systems.

  In Unix and Linux socket programming, **I/O multiplexing** is a key technique used to handle multiple sockets efficiently. The three main mechanisms available are:
  
  - **select()**
  - **poll()**
  - **epoll()** (Linux)
  - **kqueue()** (FreeBSD, macOS, etc.)
  
  ---
  
  ## ✅ 1. **select()**
  
  ### **Overview**
  - **select()** is one of the oldest I/O multiplexing functions available on Unix-like systems.
  - It allows a program to monitor multiple file descriptors (e.g., sockets) to see if they are ready for reading, writing, or have an exception condition.
  
  ### **Key Features**
  - **File descriptor limit** (typically 1024, but can be increased).
  - **Non-blocking** — it does not block the program, but it is **slow** because it copies data between kernel and user space.
  - **Returns a list of file descriptors that are ready**.
  - **Platform support** — available on **all Unix-like systems**.
  
  ### **Drawbacks**
  - **High overhead** — every call to `select()` requires a full copy of the file descriptor set.
  - **Scalability issues** — performance degrades with large numbers of file descriptors.
  
  ---
  
  ## ✅ 2. **poll()**
  
  ### **Overview**
  - **poll()** is an improved version of `select()` and is more flexible.
  - It uses a `struct pollfd` array to monitor multiple file descriptors.
  
  ### **Key Features**
  - **No hard limit on the number of file descriptors** (limited by system resources).
  - **More efficient** than `select()` in some cases.
  - **Still has the same overhead** as `select()` — data must be copied between kernel and user space.
  - **Platform support** — available on **all Unix-like systems**.
  
  ### **Drawbacks**
  - **Still has the same copy overhead** as `select()`.
  - **Not as efficient as epoll or kqueue** for large-scale applications.
  
  ---
  
  ## ✅ 3. **epoll()** (Linux)
  
  ### **Overview**
  - **epoll()** is a Linux-specific I/O multiplexing mechanism.
  - It is **much more efficient** than `select()` and `poll()` for handling large numbers of file descriptors.
  
  ### **Key Features**
  - **Two modes**: **LT (Level Triggered)** and **ET (Edge Triggered)**.
  - **No need to copy the entire set of file descriptors** each time.
  - **Efficient for large-scale applications** — handles thousands of connections efficiently.
  - **Uses a file descriptor** (e.g., `epoll_fd`) to track events.
  - **Platform support** — **Linux only**.
  
  ### **Key Advantages**
  - **Zero-copy** (in some implementations).
  - **Scalable** — can handle tens of thousands of file descriptors.
  - **More efficient** than `select()` and `poll()`.
  
  ### **Example Usage**
  ```c
  int epoll_fd = epoll_create(10);
  struct epoll_event event;
  event.events = EPOLLIN;
  event.data.fd = sockfd;
  epoll_ctl(epoll_fd, EPOLL_CTL_ADD, sockfd, &event);
  
  while (1) {
      int nfds = epoll_wait(epoll_fd, events, MAX_EVENTS, -1);
      for (int i = 0; i < nfds; ++i) {
          // handle event on events[i].data.fd
      }
  }
  ```

  The **CAP Theorem**, also known as **Brewer's Theorem**, is a fundamental concept in distributed computing that describes the trade-offs between three desirable properties of a distributed system:
  
  1. **Consistency (C)**: All nodes in the system see the same data at the same time. Once a write is successfully completed, all subsequent reads will return the latest value.
  2. **Availability (A)**: Every request receives a response, without guarantee that the response will contain the most recent version of the data.
  3. **Partition Tolerance (P)**: The system continues to operate correctly even when some messages are lost or delayed due to network partitions.
  
  According to the CAP theorem, in any distributed system that experiences network partitions (which is almost always the case), it is **impossible to guarantee all three** of these properties simultaneously. Therefore, a system must choose **two out of the three**:
  
  - **CP (Consistent and Partition Tolerant)**: Ensures consistency and partition tolerance, but may sacrifice availability.
  - **AP (Available and Partition Tolerant)**: Ensures availability and partition tolerance, but may sacrifice consistency.
  - **CA (Consistent and Available)**: Ensures consistency and availability, but cannot tolerate network partitions.
  
  ---
  
  ### Example of a **CP System**:
  A **CP system** prioritizes **consistency** and **partition tolerance**, but may **not always be available** during a network partition.
  
  **Example: A distributed database like Apache Cassandra with strong consistency enabled.**
  
  - If a network partition occurs, the system may **refuse to serve requests** until the partition is resolved to ensure that all nodes have the same data.
  - This ensures **consistency** (C) and **partition tolerance** (P), but may **reduce availability** (A) during the partition.
  
  ---
  
  ### Example of an **AP System**:
  An **AP system** prioritizes **availability** and **partition tolerance**, but may **not always provide strong consistency**.
  
  **Example: A distributed key-value store like Amazon DynamoDB or Apache Cassandra with eventual consistency.**
  
  - During a network partition, the system continues to **accept and process requests**, even if some nodes are not in sync.
  - This ensures **availability** (A) and **partition tolerance** (P), but **consistency is eventually achieved** (eventual consistency), not immediately.
  
  ---
  
  ### Summary:
  | System Type | Consistency (C) | Availability (A) | Partition Tolerance (P) |
  |-------------|------------------|------------------|--------------------------|
  | **CP**      | ✅               | ❌               | ✅                       |
  | **AP**      | ❌               | ✅               | ✅                       |

  - **Social media platforms** prioritize **availability** (A) to ensure users can always post and view content, even during network issues, so they may use **AP** systems.
  
  The CAP theorem helps architects make informed decisions about trade-offs in distributed systems based on the specific requirements of their application.

  In an operating system, both **processes** and **threads** are fundamental units of execution, but they have key differences in terms of **independence**, **resource allocation**, and **execution behavior**.
  
  ---
  
  ### ✅ **1. Definition**
  
  - **Process**:  
    A process is an instance of a program in execution. It includes the program code, its current activity (data, stack, heap), and the resources allocated to it (like memory, open files, and CPU time).
  
  - **Thread**:  
    A thread is a **lightweight** subunit of a process. It is the smallest unit of execution within a process. Threads share the same memory and resources as other threads in the same process.
  
  ---
  
  ### ✅ **2. Independence**
  
  - **Process**:  
    Processes are **independent** of each other. One process cannot directly access the memory or resources of another process without inter-process communication (IPC).
  
  - **Thread**:  
    Threads are **dependent** on each other within the same process. All threads in a process share the same memory space and resources (like open files, environment variables, etc.).
  
  ---
  
  ### ✅ **3. Resource Allocation**
  
  - **Process**:  
    Each process has its **own separate memory space** (address space), file descriptors, and other system resources. Creating a new process is more resource-intensive.
  
  - **Thread**:  
    Threads share the **same memory space** and resources of the parent process. Creating a new thread is **less resource-intensive** than creating a new process.
  
  ---
  
  ### ✅ **4. Context Switching**
  
  - **Process**:  
    Context switching between processes is **more expensive** because it requires saving and restoring the entire process state (including memory, registers, etc.).
  
  - **Thread**:  
    Context switching between threads is **cheaper** because they share the same memory and resources. Only the thread's register state needs to be saved and restored.
  
  ---
  
  ### ✅ **5. Scheduling**
  
  - **Process**:  
    The operating system schedules processes. Each process gets a slice of CPU time.
  
  - **Thread**:  
    Threads are **scheduled within a process**. Threads are typically scheduled in the same order as the process and are often managed by the thread library (like POSIX threads or Java threads).
  
  ---
  
  ### ✅ **6. Communication**
  
  - **Process**:  
    Processes communicate through **inter-process communication (IPC)** mechanisms like pipes, sockets, shared memory, or message queues.
  
  - **Thread**:  
    Threads can communicate directly with each other via **shared memory**, making communication faster and easier.
  
  ---
  
  ### ✅ **7. Use Cases**
  
  - **Processes**:
    - Used for **isolated tasks** where security or stability is important.
    - Easier to debug and manage.
    - Suitable for tasks that don’t require tight coordination.
  
  - **Threads**:
    - Used for **parallel execution** within a single program.
    - Ideal for **concurrent tasks** (e.g., a web server handling multiple requests at once).

  A **Bloom Filter** is a probabilistic data structure that is used to test whether an element is a member of a set. It is designed to answer the question: *"Is this element in the set or not?"* However, it may return **false positives** (i.e., it might incorrectly indicate that an element is present in the set when it is not), but it **never returns false negatives** (i.e., if it says an element is not in the set, it is definitely not there).
  
  ---
  
  ## 🔍 How a Bloom Filter Works
  
  1. **Initialization**:
     - A Bloom filter consists of a **bit array** of size `m`, initialized to all **0s**.
     - A set of `k` **hash functions** are used to map elements to positions in the array.
  
  2. **Adding an Element**:
     - When an element is added to the set:
       - The element is passed through each of the `k` hash functions.
       - Each hash function produces a **position** in the bit array.
       - The corresponding bits at these positions are set to **1**.
  
  3. **Checking Membership**:
     - To check if an element is in the set:
       - The element is passed through the same `k` hash functions.
       - The positions are computed, and the bits at those positions are checked.
       - If **any** of the bits is **0**, the element is **not** in the set.
       - If **all** bits are **1**, the element **might** be in the set (false positive).
  
  ---
  
  ## 📌 Key Characteristics
  
  - **Space Efficiency**: Bloom filters use much less memory than other data structures that store the actual elements.
  - **Time Efficiency**: Operations (insert and check) are **O(k)**, where `k` is the number of hash functions.
  - **False Positives**: Possible, but can be minimized by choosing appropriate `m` (bit array size) and `k` (number of hash functions).
  - **No False Negatives**: If the filter says an element is not in the set, it definitely isn't.

  ### ✅ **How a Bloom Filter Works**
  
  1. **Initialization**:
     - A Bloom filter is represented by a bit array of length `m`, initially all set to 0.
     - A set of `k` **hash functions** are used to map elements to positions in the bit array.
  
  2. **Adding an Element**:
     - When an element is added to the set:
       - The element is passed through each of the `k` hash functions.
       - Each hash function returns an index in the bit array.
       - The corresponding bits at those indices are set to 1.
  
  3. **Checking for an Element**:
     - To check if an element is in the set:
       - The element is passed through the same `k` hash functions.
       - The corresponding bits are checked.
       - If **any** of the bits is 0, the element is **not in the set** (with certainty).
       - If **all** the bits are 1, the element **might be** in the set (with a possibility of a false positive).
  
  ---
  
  ### 🔍 **Key Properties**
  
  - **Space Efficiency**: Bloom filters use significantly less memory than traditional data structures like hash tables or binary search trees.
  - **Time Efficiency**: Both insertion and lookup operations are **O(k)**, where `k` is the number of hash functions.
  - **False Positives**: There is a small probability of false positives (i.e., the filter says an element is in the set when it is not).
  - **No False Negatives**: If the Bloom filter says an element is **not** in the set, it is definitely not in the set.
  
  ---
  
  ### 📌 **Primary Use Cases in Distributed Systems**
  
  Bloom filters are particularly useful in **distributed systems** and **networked environments** where efficiency, scalability, and latency are critical. Some common use cases include:
  
  1. **Caching and Redundancy Avoidance**:
     - Used in **caching systems** (e.g., Redis, Memcached) to quickly determine if a key is already cached, avoiding unnecessary lookups.
     - Avoid redundant data transfers in distributed systems by checking if data is already present.
  
  2. **Network Communication Optimization**:
     - In **peer-to-peer (P2P) networks**, Bloom filters are used to efficiently check if a node or file is present in a network.
     - They help reduce the number of unnecessary messages by filtering out non-existent data or nodes.
  
  3. **Database Systems**:
     - Used in **database indexing** to quickly determine if a record exists, reducing the need for full disk I/O.
     - In **distributed databases**, Bloom filters can be used to optimize query routing and reduce network traffic.
  
  4. **Web Crawling and Indexing**:
     - Web crawlers use Bloom filters to keep track of which URLs have already been visited, avoiding redundant downloads and saving bandwidth.
  
  5. **Security and Authentication**:
     - Used in **rate limiting and access control** to quickly determine if a request is valid or has been seen before.
     - In **firewalls**, to quickly identify malicious or known traffic patterns.
  
  6. **Distributed Data Storage**:
     - In systems like **Apache Cassandra**, Bloom filters are used to optimize read operations by quickly determining if a row exists in a data partition.
  
  ---
  
  ### 📌 **Limitations**
  
  - **Cannot Remove Elements**: Once a bit is set to 1, it cannot be reset without potentially losing information.
  - **False Positives**: Although rare, they can occur, which may lead to unnecessary operations or data retrieval.
  - **Fixed Size**: The size of the Bloom filter is fixed, and it cannot be dynamically resized without rehashing.
  
  ---
  
  ### 📌 **Conclusion**
  
  A Bloom filter is a powerful and efficient data structure that enables fast membership queries with minimal memory usage. It is especially valuable in **distributed systems** where reducing network traffic, avoiding redundant computations, and optimizing resource usage are critical. While it has limitations (like false positives and immutability), its efficiency and scalability make it an essential tool in modern distributed computing and networking.

It is designed to handle sequential data (like text) efficiently by using **self-attention** mechanisms, which allow the model to dynamically weigh the importance of different words in a sentence. The transformer block typically consists of two main components:
  
  ---
  
  ## 🧩 1. **Self-Attention Mechanism (Multi-Head Attention)**
  
  ### Purpose:
  - Allow each position in the sequence to attend to all positions in the previous layer, capturing **long-range dependencies** and **contextual relationships**.
  
  ### Structure:
  
  #### a. **Query (Q), Key (K), and Value (V) Matrices**
  - Each input vector is projected into three different representations: **Query**, **Key**, and **Value** using learned weight matrices.
  - These projections are used to compute attention scores.
  
  #### b. **Attention Scores**
  - The attention scores are computed by taking the dot product of the **Query** and **Key** vectors, then applying a **softmax** function to normalize them.
  - These scores represent the importance of each word in the sequence relative to the current word.
  
  #### c. **Weighted Sum of Values**
  - The attention scores are used to compute a weighted sum of the **Value** vectors, resulting in a new representation of the input sequence that reflects the attention mechanism.
  
  #### d. **Multi-Head Attention (MHA)**
  - Instead of using a single attention head, the model uses **multiple attention heads** (typically 8 or more), each with its own set of Q, K, V matrices.
  - The outputs from all heads are concatenated and projected back to the original dimension.
  - This allows the model to capture different types of relationships in the input.
  
  #### e. **Residual Connection and Layer Normalization**
  - After the attention operation, a **residual connection** is added (input + attention output), followed by **layer normalization** to stabilize training.
  
  ---
  
  ## 🧩 2. **Feed-Forward Neural Network (FFN)**
  
  ### Purpose:
  - Apply a non-linear transformation to each position independently, allowing the model to learn complex patterns.
  
  ### Structure:
  
  #### a. **Position-wise Feed-Forward Network**
  - A two-layer feed-forward network is applied to each position in the sequence.
  - The first layer is a linear transformation with a ReLU activation, followed by a second linear transformation.
  - The dimensions are typically larger than the input size (e.g., from 512 to 2048).
  
  #### b. **Residual Connection and Layer Normalization**
  - Similar to the attention layer, the output of the FFN is added to the input (residual connection), followed by layer normalization.

     - Apply the position-wise feed-forward network with residual connection and layer normalization.

     - The output of the transformer block is the result of the two layers (attention and FFN), with residual connections and normalization at each step.
